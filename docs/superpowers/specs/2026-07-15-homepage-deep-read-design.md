# 홈페이지 심층읽기 (딥리드) v1 — 설계

날짜: 2026-07-15 · 승인: 대표 (브레인스토밍 + 레드팀 3라운드 통과)

## 목적

확인된 브랜드 홈페이지는 정보의 보고다 (예: horakpotato.com — /about에 미션·가치·누적 참여자 155명·프로그램 35개, /42에 파트너). 현행 Gemini grounding + 네이버 크롤은 이걸 대부분 놓친다. **사용자가 링크 확인 단계에서 "맞아요"라고 확약한 홈페이지를 서버가 직접 순회해 읽고, 그 발췌를 기존 생성 호출에 합류시켜** 한줄소개·자세히소개·활동·파트너·숫자 블록의 품질을 끌어올린다.

**하드 제약: Gemini 호출 수 불변.** 페이지 읽기는 우리 서버의 일반 HTTP fetch(API 비용 0). 추가 비용은 기존 생성 호출의 입력 토큰 증가분뿐 (~1~4원/회, flash-lite 기준. grounding 콜의 1/100 이하).

## 범위 (v1)

- **In**: 확인된 홈페이지 딥리드 → 생성 품질 강화 (소개글 + 구조 블록).
- **Out**: 칩 반영 (링크 확인이 칩 선택 뒤 + 파서 구조상 원문 텍스트로는 칩이 안 나옴 — 실험으로 확인. v2에서 위저드 순서 재검토와 함께), 인스타그램 (로그인 장벽 + 데이터센터 IP 차단 — v1 제외를 대표에게 고지 완료), 발췌 내용의 위저드 노출 UI (v1.1에서 로그 품질 보고 검토).

## 흐름

```
LinkPicker "맞아요"(또는 직접입력) → 위저드가 확정 URL을 생성 요청에 포함
→ 생성 요청 처리 중 서버가 홈페이지 순회 (Gemini 호출과 병렬, 예산 8초)
→ 프롬프트에 [홈페이지 발췌] 블록 합류 → 생성
→ 실패(차단·타임아웃·JS전용) 시 발췌 블록 없이 기존 동작 그대로 (조용한 저하)
```

- **프리페치 없음 (결정)**: Vercel 서버리스는 인스턴스 간 메모리 비공유 + Supabase 스키마 변경 금지 관례 → 공유 캐시 테이블 없이 **생성 요청 안에서 fetch**. 생성엔 Gemini 콜(수 초~십수 초)이 이미 있으므로 홈페이지 읽기를 그와 병렬로 돌려 체감 지연 최소화. 클라이언트는 **URL만** 전달하고 발췌 텍스트는 서버에서만 생성 (클라이언트 주입 차단). 별도 fetch 엔드포인트가 없으므로 fetch-any-URL 남용 표면도 없음.

## 읽기 엔진 (`src/lib/homepage.ts` 신규)

1. **홈 fetch** → 원본 HTML에서 same-host `<a>` 링크 수확 (본문 추출로 nav를 지우기 **전**에).
2. **후보 점수화**: 경로·앵커텍스트 키워드 (about/story/company/intro/소개/스토리 > 활동/프로그램/program/news/market > 파트너/협업/partner/collab > 기타). 상위 최대 9개 선택 (홈 포함 총 10페이지).
3. **병렬 fetch**: 전체 하드 예산 ~8초, 늦는 페이지 포기. per-page 타임아웃 4초.
4. **본문 추출**: script/style/nav/header/footer/aside 제거 → 텍스트화. 추출 **후** per-page 자름 (~3.5k자). 인코딩: Content-Type/meta charset 감지 (EUC-KR 레거시 카페24 대응).
5. **JS 전용 사이트 폴백** (대표 요청 "모든 형태"): 본문이 빈약해도 `<title>`, `og:title/description/site_name`, `meta description`, **JSON-LD**(Organization/LocalBusiness 등)는 정적 HTML에 있으므로 이것만이라도 건짐.
6. **라벨링**: 각 발췌에 메뉴 앵커텍스트 라벨 `[홈페이지—파트너 감자]` (≤20자, 텍스트만, 중복 제거, 정크면 URL 슬러그 폴백). 프롬프트엔 "라벨은 참고용" 명시.
7. **총량 상한 ~18k자**, 우선순위 순 트림 (소개 계열 최우선).

출력: `{ digest: string, pages: number, ok: boolean, failReason?: string }` — digest가 비면 프롬프트에 빈 헤더도 넣지 않음.

## 안전 (레드팀 통과 조건)

- **SSRF**: http(s)만, 사설·링크로컬·메타데이터 IP 대역(IPv4+IPv6) 차단, 리다이렉트마다 재검증, 리다이렉트 상한 3, content-type text/html만, 스트리밍 중 크기 상한 1MB, 정직한 UA(`collab5-bot`).
- **robots.txt**: 사용자 확약 홈 1장은 항상 허용(사용자의 대리인 자격), 하위 페이지는 disallow 존중. robots fetch는 홈 fetch와 병렬.
- **스킵 리스트**: blog.naver.com·smartstore·modoo·인스타·페북 등 → 시도 안 함 (기존 동작 유지).
- **확약 문구**: LinkPicker 확인 카피를 "우리 브랜드의 공식 홈페이지가 맞아요"로 — 남의 사이트 확약 리스크 완화 (대표 veto 지점, 배포 후 고지).
- **레이트리밋**: 생성 요청 자체가 기존 흐름의 일부라 별도 엔드포인트 없음. 동일 URL 반복 생성은 요청 내 1회 fetch로 자연 제한.

## 프롬프트 (기존 생성 호출에 합류)

- draft2(구조화)·options(5지선다) 프롬프트에 `[홈페이지 발췌]` 블록 추가 + 규칙: **홈페이지의 사실(숫자·미션·파트너명·활동명·연혁)을 조사메모보다 우선 신뢰하고 적극 반영하되, 문장 그대로 베끼기 금지. 라벨은 참고용 힌트.**
- researchMemo는 **불변** (칩 파서·thin 판정·재크롤 방지·칩 DB 백로그와 간섭 없음). 발췌는 프롬프트 조립 시에만 합류하는 별도 필드.
- 프롬프트 대변경 → 배포 전 **Gemini 교차검증 1회** (관례, 상시 호출 수는 불변).

## 검증

- 픽스처 유닛테스트: 아임웹 정상 2~3 + EUC-KR 카페24 + JS 전용(깨끗한 빈 digest 단언) + 대용량(크기 상한) + 리다이렉트.
- E2E: horakpotato.com으로 위저드 전체 흐름 (dev 서버 DOM 검증).
- tsc + build 통과 후 배포.
- **배포 후 로깅**: 호스트 클래스 + 실패 사유(dns/timeout/403/robots/non-html) → **1주 뒤 성공률 리포트** (아임웹·카페24 계열 우선 지원임을 대표 인지).

## 결정 로그

- 프리페치·공유 캐시 제거 (서버리스 메모리 비공유 + 스키마 불변 관례 → 생성 요청 내 병렬 fetch로 단순화. 레드팀 사인오프 조건인 "캐시 저장 위치 결정" 충족)
- 최대 10페이지 (대표: "최소 4 최대 10" — 병렬이라 4장과 속도 동일, 총량 트림으로 노이즈 방지)
- 조용한 반영 (v1은 UI 노출 없음 — 대표 확인: "홈페이지 기준으로 소개서를 작성해주는 것")

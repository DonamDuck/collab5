# collab-ab-test — 로직 업그레이드 A/B 방법론

같은 업체로 소개서 2개(A/B)를 만들어 비교함으로써 크롤·생성 파이프라인의
"부족한 지점"을 소스 단위로 역추적하고, 코드/프롬프트 개선안을 도출하는 실험 절차.

핵심 질문: **"사람이 리서치로 보강하면 뭐가 좋아지는가? 그 재료는 파이프라인의
어느 단계에서 유실됐는가?"**

## 실험 설계 원칙

- **A = 베이스라인**: 지금 제품이 **혼자** 만들 수 있는 최고치. 사람 리서치 개입 0.
- **B = 보강**: A와 동일한 크롤 결과 위에 Claude의 직접 리서치를 얹은 버전.
- ⚠️ **B에서 크롤을 다시 돌리지 않는다.** 크롤에는 랜덤성이 있어 재실행하면
  "보강 효과"와 "크롤 랜덤성"이 섞여 비교가 오염된다. 변수는 오직 "사람 보강" 하나여야 한다.
- 유료 크롤(Gemini)은 **대표가 명시적으로 요청할 때만** 실행. A버전 생성 시 1회가 기본이며,
  그 이상의 크롤 호출은 하지 않는다.

## 절차

### 0. 사전 준비 — slug 확보 (중요, 아래 "slug 주의사항" 필독)

- 브라우저는 **대표 계정으로 로그인된 세션**에서 진행한다.
  비밀번호는 에이전트가 절대 다루지 않는다 — 로그인이 필요하면 대표가 직접 입력.
- 상호는 **순한글(ASCII 문자·숫자 없이)**로 입력해 `m-xxxxxx` 형태 slug를 받는다.

### 1. A버전 (베이스라인) 생성

prod `https://collab5.vercel.app/register`의 AI 위저드에서:

1. **상호·지역·업종만** 입력 (상호는 순한글 — slug 주의사항 참조)
2. 크롤이 제안하는 **칩 전부 선택**
3. 별표(확신도) **4개**
4. 링크 확인 (크롤이 찾은 인스타·홈피 등이 실제 업체 것인지)
5. 한줄 소개·자세히 소개 **후보 중 최선 1개 선택** (수정·가필 금지)
6. 저장

원칙: 선택지 안에서 고르는 것까지만 허용, **텍스트를 쓰거나 고치는 개입은 0**.
이게 "지금 제품이 혼자 만들 수 있는 최고치"다. 저장 후 `/m/{slug}` URL을 기록한다.

### 2. B버전 (보강) 생성

1. A와 **동일한 베이스**에서 시작한다 (크롤 재실행 금지 — A 생성 시의 크롤 결과를 그대로 사용).
   실무적으로는 A 저장 직후 같은 위저드 상태에서 이어가거나, A를 복제한 뒤 수정 화면에서 보강한다.
2. Claude가 브라우저 탭으로 **직접 리서치**:
   - 인스타그램 바이오·포스트 (사장님이 직접 쓴 텍스트)
   - 보도기사
   - 네이버 블로그 (사장님 운영 블로그 우선)
3. 리서치에서 얻은 **사장님 직접 작성 텍스트**를 근거로 각 섹션을 수동 보강.
   창작 금지 — 소스에 있는 사실만 반영하고, 각 보강 항목의 출처를 메모해 둔다 (3단계 비교표 재료).
4. 상호에 구분자를 붙여 저장 (예: "캔앤코르크B" — slug는 이미 고정돼 있으므로 안전).

### 3. 비교표 작성

섹션별로 A vs B를 표로 기록한다:

| 섹션 | A (베이스라인) | B (보강) | 차이 | 차이의 출처 |
|---|---|---|---|---|
| 한줄 소개 | | | | (예: 인스타 바이오) |
| 자세히 소개 | | | | |
| 활동 | | | | |
| 콜라보 | | | | |
| 소개된 곳 | | | | |
| 공간 | | | | |
| 키워드 | | | | |

"차이의 출처"가 핵심 — B 우위 재료가 **어떤 소스**(인스타/기사/블로그)에서 왔는지 반드시 기록.

### 4. 로직 업그레이드 도출

B 우위 재료마다 역추적한다:

1. 그 소스를 파이프라인이 **아예 안 봤나?** (검색 쿼리·크롤 대상 문제)
2. 봤는데 **추출 단계에서 버렸나?** (파싱·요약 프롬프트 문제)
3. 추출됐는데 **생성 단계에서 안 썼나?** (생성 프롬프트·후보 선택 문제)

유실 지점별로 코드/프롬프트 개선안을 **우선순위 목록**으로 정리한다.
관련 코드: `src/lib/enrich.ts` (크롤·생성 파이프라인).
AI 프롬프트를 크게 바꿀 땐 Gemini 교차검증 관례를 따르되, 유료 호출 수는 늘리지 않는다.

### 5. 정리 (라운드 2 & 청소)

- 개선을 적용한 뒤, **같은 업체로 C버전**을 만들어 **B와 재비교** (라운드 2).
  C가 B에 근접하면 로직 업그레이드가 성공한 것.
- 최종적으로 **최고 버전 하나만 남긴다.** 나머지(A버전 등 prod 테스트 데이터)의
  **삭제는 대표가 직접** 한다 — 에이전트가 prod 데이터를 삭제하지 않는다.

## slug 주의사항 (2026-07-21 실측)

slug 생성 로직은 `src/lib/actions.ts`의 `slugify()` (links.ts 아님):

```ts
/** 이름 → slug. 한글 등 비ASCII면 랜덤 핸들로 폴백(mock 단계). */
function slugify(name: string): string {
  const ascii = name
    .toLowerCase()
    .trim()
    .replace(/[^a-z0-9]+/g, "-")
    .replace(/^-+|-+$/g, "");
  return ascii || "m-" + Math.random().toString(36).slice(2, 8);
}
```

동작:
- 상호에서 **ASCII 영문·숫자만 남기고** 나머지(한글 포함)는 전부 `-`로 바뀐 뒤 트림된다.
- **실측 함정**: "캔앤코르크A" / "캔앤코르크B"로 입력하면 한글이 전부 탈락하고
  slug가 `/m/a`, `/m/b`가 된다. 숫자도 마찬가지 ("캔앤코르크2" → `/m/2`).
  이런 한 글자 slug는 충돌 위험이 크고 (createMaker에 중복 slug 방지 로직 없음) URL도 지저분하다.
- **순한글 상호**(ASCII 문자·숫자 0개)면 ascii가 빈 문자열 → `m-` + 랜덤 6자
  폴백으로 `m-osmjuk` 같은 slug가 나온다. 이게 원하는 형태.

**지침: 반드시 `m-xxxxxx` 형태 slug로 생성되게 하라.**

1. 위저드에는 상호를 **순한글로만** 입력한다 (영문·숫자 한 글자도 넣지 않는다).
   → `m-` 랜덤 slug를 받는다.
2. A/B 구분은 저장 **후** 수정 화면에서 상호에 A/B를 붙인다.
   `updateMakerContent`는 slug를 변경 대상에서 제외하므로(`Omit<..., "slug">`)
   상호를 "캔앤코르크A"로 바꿔도 slug는 `m-xxxxxx`로 고정 유지된다.
3. 각 버전의 `/m/{slug}` URL을 생성 즉시 기록해 둔다 (랜덤이라 다시 유추 불가).

## 브라우저 자동화 함정 (실측)

- `read_page`의 ref는 **뷰포트 안에 있을 때만 유효**하다. 스크롤 후엔 다시 read_page.
- React 상태 배칭 때문에 **한 번의 JS 실행에서 여러 클릭을 하면 마지막 것만 반영**된다.
  칩·키워드 선택은 **한 번에 하나씩** 클릭한다.
- 클릭 후 화면이 안 바뀐 것처럼 보이면 `screenshot`으로 **강제 페인트** 후 `read_page`로 상태 확인.
- **최종 저장 버튼은 ref 실클릭**으로 누른다 (JS dispatch 금지).
- 저장은 로그인된 대표 계정 세션에서만. 비밀번호 입력이 필요한 상황이 오면 중단하고 대표에게 요청.

## 금지 사항 요약

- 대표 요청 없는 유료 크롤(Gemini) 호출 금지
- A버전에 텍스트 개입 금지 (선택만 허용)
- 에이전트의 비밀번호 취급 금지
- 에이전트의 prod 테스트 데이터 삭제 금지 (대표가 직접)

# collab-ab-test — 로직 업그레이드 A/B 방법론

같은 업체로 소개서 2개(A/B)를 만들어 비교함으로써 크롤·생성 파이프라인의
"부족한 지점"을 소스 단위로 역추적하고, 코드/프롬프트 개선안을 도출하는 실험 절차.

핵심 질문: **"사람이 리서치로 보강하면 뭐가 좋아지는가? 그 재료는 파이프라인의
어느 단계에서 유실됐는가?"**

## 실험 설계 원칙

- **A = 베이스라인**: 지금 제품이 **혼자** 만들 수 있는 최고치. 사람 리서치 개입 0.
- **B = 보강**: A와 동일한 크롤 결과 위에 Claude의 직접 리서치를 얹은 버전.
- ⚠️ **B에서 크롤을 다시 돌리지 않는다.** 크롤에는 랜덤성이 있어 재실행하면
  "보강 효과"와 "크롤 랜덤성"이 섞여 비교가 오염된다. 변수는 오직 "사람 보강" 하나여야 한다.
- 유료 크롤(Gemini)은 **대표가 명시적으로 요청할 때만** 실행. A버전 생성 시 1회가 기본이며,
  그 이상의 크롤 호출은 하지 않는다.

## 절차

### 0. 사전 준비

- 브라우저는 **대표 계정으로 로그인된 세션**에서 진행한다.
  비밀번호는 에이전트가 절대 다루지 않는다 — 로그인이 필요하면 대표가 직접 입력.
- 상호에 A/B/C를 붙여도 slug는 안전하다(2026-07-22 가드 배포 — "slug" 절 참조).
- **라운드당 유료 콜 예산 = 2콜**(키워드 조사 1 + 생성 1). 실패해도 위저드를 처음부터
  다시 돌리지 말고, 실패 지점과 화면 상태를 그대로 보고한다.

### 1. A버전 (베이스라인) 생성

prod `https://collab5.vercel.app/register`의 AI 위저드에서:

1. **상호·지역·업종만** 입력
2. 크롤이 제안하는 **칩 전부 선택**
3. 별표(확신도) **4개**
4. 링크 확인 (크롤이 찾은 인스타·홈피 등이 실제 업체 것인지)
5. 한줄 소개·자세히 소개 **후보 중 최선 1개 선택** (수정·가필 금지)
6. 저장

원칙: 선택지 안에서 고르는 것까지만 허용, **텍스트를 쓰거나 고치는 개입은 0**.
이게 "지금 제품이 혼자 만들 수 있는 최고치"다. 저장 후 `/m/{slug}` URL을 기록한다.

### 2. B버전 (보강) 생성

1. A와 **동일한 베이스**에서 시작한다 (크롤 재실행 금지 — A 생성 시의 크롤 결과를 그대로 사용).
   실무적으로는 A 저장 직후 같은 위저드 상태에서 이어가거나, A를 복제한 뒤 수정 화면에서 보강한다.
2. Claude가 브라우저 탭으로 **직접 리서치**:
   - 인스타그램 바이오·포스트 (사장님이 직접 쓴 텍스트)
   - 보도기사
   - 네이버 블로그 (사장님 운영 블로그 우선)
3. 리서치에서 얻은 **사장님 직접 작성 텍스트**를 근거로 각 섹션을 수동 보강.
   창작 금지 — 소스에 있는 사실만 반영하고, 각 보강 항목의 출처를 메모해 둔다 (3단계 비교표 재료).
4. 상호에 구분자를 붙여 저장 (예: "캔앤코르크B" — slug는 이미 고정돼 있으므로 안전).

### 3. 비교표 작성

섹션별로 A vs B를 표로 기록한다:

| 섹션 | A (베이스라인) | B (보강) | 차이 | 차이의 출처 |
|---|---|---|---|---|
| 한줄 소개 | | | | (예: 인스타 바이오) |
| 자세히 소개 | | | | |
| 활동 | | | | |
| 콜라보 | | | | |
| 소개된 곳 | | | | |
| 공간 | | | | |
| 키워드 | | | | |

"차이의 출처"가 핵심 — B 우위 재료가 **어떤 소스**(인스타/기사/블로그)에서 왔는지 반드시 기록.

### 4. 로직 업그레이드 도출

B 우위 재료마다 역추적한다:

1. 그 소스를 파이프라인이 **아예 안 봤나?** (검색 쿼리·크롤 대상 문제)
2. 봤는데 **추출 단계에서 버렸나?** (파싱·요약 프롬프트 문제)
3. 추출됐는데 **생성 단계에서 안 썼나?** (생성 프롬프트·후보 선택 문제)

유실 지점별로 코드/프롬프트 개선안을 **우선순위 목록**으로 정리한다.
관련 코드: `src/lib/enrich.ts` (크롤·생성 파이프라인).
AI 프롬프트를 크게 바꿀 땐 Gemini 교차검증 관례를 따르되, 유료 호출 수는 늘리지 않는다.

### 5. 정리 (라운드 2 & 청소)

- 개선을 적용한 뒤, **같은 업체로 C버전**을 만들어 **B와 재비교** (라운드 2).
  C가 B에 근접하면 로직 업그레이드가 성공한 것.
- 최종적으로 **최고 버전 하나만 남긴다.** 나머지(A버전 등 prod 테스트 데이터)의
  **삭제는 대표가 직접** 한다 — 에이전트가 prod 데이터를 삭제하지 않는다.

## ⭐저장 없는 검증 라운드 (권장 — 2026-07-22 확립)

"방금 배포한 기능이 실제로 동작하나?"만 확인하면 될 때는 **저장하지 않는다.**
위저드를 끝까지 돌려 **폼까지만 진입한 뒤 DOM으로 프리필 값을 읽고 탭을 닫는다.**

- prod에 테스트 소개서가 안 쌓인다(대표가 나중에 지울 것이 0건).
- 확인할 수 있는 것: AI가 채운 모든 값(칩 ✓ 상태, textarea `.value`, 배지 유무),
  한줄·자세히 후보 전문, 활동/콜라보/공간 힌트.
- 확인할 수 **없는** 것: 저장 로직·`/m` 렌더 결과. 그건 정식 라운드에서.
- 유료 콜은 정식 라운드와 동일하게 2콜 든다(생성까지 가야 하므로).

⚠️ **placeholder를 값으로 오판하지 마라.** 회색 예시 문구는 `placeholder` 속성이다.
반드시 `javascript_tool`로 `.value`를 읽어 판정한다.

## ⭐Vercel 로그 판독 — 파이프라인이 실제로 돌았는지 확인

생성물만 보면 "재료가 안 들어온 것"과 "들어왔는데 생성이 안 쓴 것"을 구분할 수 없다.
서버 로그가 이 둘을 가른다. **vercel.com → 프로젝트 → Logs → 검색어 입력.**

| 로그 | 의미 |
|---|---|
| `[enrich] instagram digest: bio+N posts (@handle)` | 인스타 발췌가 서버에 도착함. N=캡션 수 |
| `[enrich] instagram profile fetch failed …` | 인스타 fetch 실패(차단·타임아웃) |
| `[article-excerpts] {urls:N, got:M}` | 기사 본문 발췌 성공 M건 |
| `[homepage-digest] {...}` | 홈페이지 딥리드 결과 |
| (아무것도 없음) | 그 재료가 서버까지 전달조차 안 됨 → 배선 버그 |

**판정 분기**: 로그에 재료가 있는데 생성물에 없다 → **프롬프트 문제**(반영 지시 강화).
로그 자체가 없다 → **수집·배선 문제**(fetch·전달 경로 수정).

⏰ **로그는 보관 기간이 짧다(Hobby ~1시간).** 크롤 직후 바로 확인한다.
대표에게 확인을 요청할 때도 "지금 봐달라"고 함께 알린다.

## 판정 기준 (C가 B에 근접했는지)

정성 인상 대신 셀 수 있는 지표로 판정한다:

1. **사장님 원문 표현 회수 수** — 바이오·캡션의 고유 표현(애칭·슬로건)이
   후보 몇 개에 살아 있나 (예: 한줄 5개 중 3, 자세히 6개 중 6).
2. **사실 오류 0** — 특히 크롤이 지어내기 쉬운 항목(운영 정보·연도·매체명).
3. **활동 개수와 구체성** — 개수뿐 아니라 "대표 메뉴명이 들어갔나" 수준까지.
4. **빈 필수 항목 0** — 사장님이 손댈 것 없이 저장 가능한가.
5. **대표 블라인드 리딩** — 최종 판단은 대표가 A/B/C를 나란히 읽고 고른다.

## slug — 우회 불필요 (2026-07-22 수정 배포됨)

**결론: 상호에 A/B/C를 그냥 붙여도 된다.** 아래 우회 절차는 이제 필요 없다.

과거 함정: `slugify()`가 ASCII만 남기는 탓에 "캔앤코르크A" → `/m/a`, "캔앤코르크B" → `/m/b`처럼
한 글자 slug가 나왔다(중복 slug 방지 로직도 없어 충돌 위험). 이를 `src/lib/actions.ts`에서 수정:

```ts
// ascii가 3자 미만이면 랜덤 핸들로 폴백 (2026-07-22)
return ascii.length >= 3 ? ascii : "m-" + Math.random().toString(36).slice(2, 8);
```

- 실측: "캔앤코르크C" → `m-gnd43q` ✅ (한글+영문 1자 → 폴백 정상 작동)
- 영문 상호(ASCII 3자 이상)는 그대로 그 ascii가 slug가 된다 — 정상 동작이다.
- slug는 랜덤이라 **생성 즉시 `/m/{slug}` URL을 기록**해 둔다(다시 유추 불가).
- 참고: `updateMakerContent`는 slug를 변경 대상에서 제외하므로(`Omit<..., "slug">`),
  저장 후 상호를 바꿔도 slug는 고정 유지된다.

## 브라우저 자동화 함정 (실측)

- `read_page`의 ref는 **뷰포트 안에 있을 때만 유효**하다. 스크롤 후엔 다시 read_page.
- React 상태 배칭 때문에 **한 번의 JS 실행에서 여러 클릭을 하면 마지막 것만 반영**된다.
  칩·키워드 선택은 **한 번에 하나씩** 클릭한다.
- 클릭 후 화면이 안 바뀐 것처럼 보이면 `screenshot`으로 **강제 페인트** 후 `read_page`로 상태 확인.
- **최종 저장 버튼은 ref 실클릭**으로 누른다 (JS dispatch 금지).
- 저장은 로그인된 대표 계정 세션에서만. 비밀번호 입력이 필요한 상황이 오면 중단하고 대표에게 요청.

## 금지 사항 요약

- B에서 크롤 재실행 금지 (실험 오염)
- 대표 요청 없는 유료 크롤(Gemini) 호출 금지 · 라운드당 2콜 초과 금지 (실패해도 재시도 금지)
- A버전에 텍스트 개입 금지 (선택만 허용)
- 에이전트의 비밀번호 취급 금지
- 에이전트의 prod 테스트 데이터 삭제 금지 (대표가 직접)
- `.value` 대신 placeholder를 보고 "채워졌다" 판정 금지

## 실증 기록 — 캔앤코르크 (2026-07-21~22)

이 방법론으로 실제 파이프라인이 개선된 1회차 기록. 다음 실험의 기대치 기준선으로 쓴다.

- **A**(구 로직 단독): 활동 섹션 없음, "이색 공간으로 소개됐어요" 수준의 범용 문장,
  협업 기대 빈칸, 후보에 운영 정보 오류.
- **B**(Claude 리서치 보강): 인스타 바이오·캡션에서 애칭·분위기 표현 회수, 기사에서
  대표 메뉴·이벤트명 회수 → 대표가 "완전히 더 낫다" 판정.
- **격차 역추적 결과 → 로직 10건 배포**(`b7277b4`~`719ead3`): 인스타 딥리드, 기사 딥리드,
  활동 0~5건, offersHint(필수 항목 프리필), slug 가드, 유통기한 정보 금지 등.
- **C**(신 로직 단독): 활동 3건 자동 생성, 기사 구체 사실 반영, 파트너 찾기 자동,
  운영 정보 오류 소멸 → **B에 근접 확인**. 남은 격차 = 인스타 캡션(게시물 URL 발견이 병목).
- **검증 라운드**(저장 없음): offersHint 칩 4/7 자동 + 맞춤 문장 실측, 바이오 반영 자세히 6/6.

교훈: **B 우위 재료의 대부분은 "사장님이 직접 쓴 글"이었고, 그중 서버가 자동으로
가져올 수 있는 것(바이오·기사)은 코드화에 성공했다.** 사람 개입이 아직 필요한 영역은
브라우저로만 가능한 탐색(게시물 URL 발견 등)으로 좁혀졌다.
